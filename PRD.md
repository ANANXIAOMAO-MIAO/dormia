# Dormia — 产品需求文档（PRD）

版本：v0.1（MVP）
文档目的：指导 MVP 阶段的技术开发与测试验收
产品负责人：滕琳
最后更新：2026-07-05

---

## 1. 产品背景

Dormia 源自一次关于"创造力压力"的自我对话：许多宏大的、模糊的、甚至看起来"不可能"的想法，一旦被当作"必须完成的项目"，就会立刻被自我审判压垮，最终一个都不敢写下来。

Dormia 不是效率工具，也不是待办清单。它是一个**没有期待、没有观众、没有产出义务**的容器。用户把闪现的念头随手放进来，它们不需要被整理、被执行、被验证价值。这个动作本身，就是对"必须有用"这种内在声音的一次温柔反抗。

产品的情绪基调是**梦境**，而不是"死亡"或"停尸房"：想法沉入深处，安静地漂浮，不需要醒来；但如果某天它自己"醒"了，那是意外的惊喜，而不是任务。

---

## 2. 产品定位

- **产品名称**：Dormia
- **档案页命名**：深处
- **一句话定位**：一个让宏大念头可以安然沉睡、不需要被实现的地方。
- **可选副标题 / Slogan**（用于首页或分享文案，不作为产品名）：
  - "想法可以不去任何地方。"
  - "在这里，灵感不需要变成任何东西。"
  - "深处有光，无需点燃。"

### 非目标（Non-goals）

- 不是任务管理 / 待办工具，不做提醒、截止日期、进度追踪。
- 不做社交功能（点赞、评论、公开广场）。MVP 阶段甚至不做"分享给他人查看"。
- 不追求让用户"把想法做成作品"，AI 不输出建议、不鼓励行动。
- 不做多端同步、多设备登录（MVP 阶段）。

---

## 3. 目标用户与阶段规划

- **MVP 阶段**：仅供产品负责人自己使用，验证核心体验是否成立。不做登录，数据存储在本地浏览器（localStorage）。
- **后续阶段**（非本期开发范围，仅供技术选型预留）：如果验证有效，考虑引入账号体系与云端存储（如 Neon Postgres），支持分享给他人各自使用。技术实现应尽量避免在 MVP 阶段绑死无法迁移的本地方案。

---

## 4. 信息架构

Dormia 采用极简两页结构，不设置底部导航栏、不设置多级菜单。

```
Dormia
├── 首页（输入）— 默认进入页
│   ├── 顶部：线条动画（呼吸感/ 有机形态，营造梦境氛围）
│   └── 底部：文字输入框 + 提交
│
└── 深处（档案页）
    ├── 弥散光晕背景 + 想法节点随机散布
    ├── 点击节点 → 弹出想法详情
    │   ├── 想法原文
    │   ├── AI 共情回应（一句话）
    │   ├── 编辑 / 删除 操作
    └── （无分类标签展示，聚类仅用于视觉分组/着色，不做文字标签导航）
```

两页之间通过一个轻量入口切换（如顶部图标或滑动手势），不使用"返回/新建"这类任务型语言。

## 5. 核心用户旅程

### 旅程 A：极速记录，随手关闭

1. 用户打开 Dormia，直接看到首页的输入框（无需点击"新建"）。
2. 输入几个字或一段话，点击提交。
3. 界面给出一句安静的反馈文案（见 6.3），随后输入框清空，用户可直接关闭页面。
4. 全程无需跳转、无需确认弹窗，目标是 10 秒内完成。

### 旅程 B：记录后顺手查看"深处"

1. 用户完成旅程 A 的记录动作。
2. 顺手切换到"深处"页面，看到过往想法以光晕节点的形式散布着（新想法也已自动汇入其中）。
3. 点击某个节点，查看：
   - 想法原文
   - AI 生成的一句共情回应（安静、克制，不含建议或评价）
4. 用户可以选择编辑该想法文字，或删除它。删除无需二次确认弹窗中出现"确定要删除吗"这类严肃措辞，保持基调一致（措辞见 6.3）。

## 6. 功能需求详情

### 6.1 首页 — 记录

| 项目 | 要求 |
|---|---|
| 入口 | 打开应用即所见即所得的输入框，无"新建"按钮、无空状态引导教程 |
| 输入形式 | 纯文本，不限制字数上限（但 UI 上不做多行编辑器的复杂排版，保持轻量） |
| 顶部动画 | 线条绘制的有机动态图形（非任务加载动画），持续轻微呼吸式运动，传达平静而非等待感 |
| 提交方式 | 点击提交按钮或键盘"发送"直接提交，无二次确认 |
| 提交后行为 | 显示一句反馈文案 → 输入框清空 → 用户可继续输入或直接离开 |
| 后台动作 | 提交后**静默**触发：① AI 共情回应生成 ② 自动聚类归组（见 6.2），均不阻塞用户操作，不在前台展示进度 |

### 6.2 深处 — 档案与回顾

| 项目 | 要求 |
|---|---|
| 视图形式 | 空间散布视图：每条想法是一个光晕节点，随机分布在画布上（MVP 阶段位置随机生成，不做力导向真实聚类布局） |
| 视觉分组 | 每条想法归属一个由 AI 判定的语义分组（聚类），分组以颜色/色相区分节点光晕颜色，不展示分组名称文字标签（避免"分类"带来的整理感与压力） |
| 交互 | 支持缩放/滑动浏览全部节点；点击节点弹出详情卡片 |
| 详情卡片内容 | 想法原文、AI 一句共情回应、编辑入口、删除入口 |
| 编辑 | 允许修改想法文字。编辑后是否重新触发聚类与共情回应生成，由技术实现决定，建议触发（保持数据一致性） |
| 删除 | 允许永久删除，删除即从"深处"移除，不做回收站（除非未来另有需求） |
| 排序/查找 | MVP 不做搜索、不做时间排序列表视图，只保留空间散布这一种呈现方式 |

### 6.3 文案与语气规范

所有系统文案遵循：**冷静、克制、不评判、不鼓励行动**。

| 场景 | 参考文案 |
|---|---|
| 提交成功反馈 | "它已安静地沉入深处。" |
| 空状态（深处页尚无内容时） | 建议使用类似"这里还很安静。"，不使用"快去记录第一个想法吧！"这类引导式运营文案 |
| 删除确认（如需要） | 避免"确定删除吗？此操作不可撤销"的警告式措辞，可用更轻的表达，如"它将不再被记起。" |
| AI 共情回应 | 见 6.4 |

### 6.4 AI 能力规格

Dormia 使用 DeepSeek API（Chat Completions）驱动两类后台能力，均在用户提交想法后**静默异步生成**，用户点击查看详情时内容已就绪，不出现"生成中"的等待态。

#### 6.4.1 共情回应

- **触发时机**：想法提交后台后立即生成一次；若用户编辑了想法文字，建议重新生成。
- **形式**：仅一句话（一个短句，非分段落），不包含建议、不包含行动号召、不包含情绪鼓励词（如"加油""你可以的"）。
- **语气**：安静、克制、命名式的共情——只是"看见"这个想法并简短回应它的情绪或意图，不做价值判断，不问"你打算怎么做"。
- **反例**（不应出现）：
  - "这是个好主意，要不要试试从小处开始？"（包含建议）
  - "你真棒，敢于有这样的想法！"（包含鼓励）
- **正例参考感觉**：
  - 想法："做一个关于公平正义的系统"
  - 回应示例感觉："关于秩序的渴望，被听见了。"（实际文案由 Prompt 设计阶段打磨，此处仅作语气参照，非最终文案）

#### 6.4.2 语义聚类分组

- **触发时机**：每次新想法提交后，自动对全部想法重新执行一次聚类分组。
- **技术现实与建议方案**：
  - DeepSeek 官方 API **不提供 embedding（向量）接口**，只有对话补全接口，因此"标准向量相似度聚类"无法直接用 DeepSeek 完成。
  - 建议方案二选一，供技术方案设计阶段决策：
    1. **LLM 归纳打标（推荐 MVP 用）**：每次新增想法后，把"当前想法 + 已有的分组标签列表"一起交给 DeepSeek，让模型判断该想法应归入已有的某个分组，还是需要新建一个分组，并给分组一个简短的内部标签（标签本身不展示给用户，仅用于着色分组）。实现简单、无需额外依赖，效果贴近"AI 自由归类"的诉求。
    2. **引入独立 Embedding 服务**：使用第三方 embedding 模型（如开源 BGE 系列或其他向量服务）计算文本向量，再做向量聚类（如层次聚类/KMeans）。效果更"数学化"，但需要额外的服务依赖与成本，MVP 阶段非必需。
  - 建议 MVP 阶段采用方案 1，保留后续切换到方案 2 的可能性（不要在数据结构上写死聚类方式）。
- **分组数量**：不预设固定分类体系（不设"创作/生活/事业"等预定义标签），完全由 AI 根据历史内容自由生成分组。

## 7. 视觉与交互设计指引

- **整体基调**：暗色背景，弥散光晕（misty diffuse glow），偏冷静的蓝紫色系，避免高饱和、高对比的"元气感"配色。
- **氛围参照**：宁静、深邃、略带神秘感，类似深夜、深海、梦境边界的视觉联想；避免任何"庆祝""成就感"式的视觉反馈（如彩带、勋章、进度条）。
- **首页布局**：上半部分为线条动画区域（有机、流动、非几何图形的呼吸感动效），下半部分为输入框，页面整体保持大量留白。
- **深处页布局**：参考弥散光晕节点散布的视觉风格（节点大小可随想法长度或新旧程度有细微差异，但不做强烈的数据可视化感）。
- **动效原则**：所有过渡动画应缓慢、柔和，避免弹跳、快速缩放等"轻快"感的效果，符合"沉入"的物理直觉（例如新想法生成时可有一个缓慢下沉/扩散的入场动效）。
- **字体与排版**：简洁、留白充足，避免密集排版和强调色的滥用。

## 8. 数据模型（建议）

MVP 阶段数据存储于浏览器 localStorage，字段设计需考虑未来平滑迁移到云端数据库，建议结构：

```
Idea {
  id: string              // 唯一标识
  text: string             // 想法原文
  createdAt: timestamp
  updatedAt: timestamp
  clusterId: string | null // 所属分组（AI 生成）
  aiResponse: string        // AI 共情回应（一句话）
  aiResponseGeneratedAt: timestamp | null
  position: { x: number, y: number } // 深处页随机散布坐标，生成后固定存储，避免每次刷新位置跳动
}

Cluster {
  id: string
  label: string     // AI 内部标签，不直接展示给用户
  color: string     // 用于节点光晕着色
  createdAt: timestamp
}
```

## 9. 技术方案建议（非强制，供开发参考）

- **形态**：H5 Web App（非微信小程序），原因：视觉自由度需求高（自定义光晕、动效），且不希望被小程序审核与框架能力所限制；后续如需在微信内传播，可用小程序 WebView 或 H5 分享链接承载。
- **可选加强**：支持 PWA（添加到主屏幕、全屏无浏览器栏），提升"类 App"体验，MVP 阶段可作为加分项而非必须项。
- **AI 接口**：DeepSeek Chat Completions API，用于共情回应生成与语义归类打标（见 6.4）。
- **数据存储**：MVP 用 localStorage，不接入登录与云数据库；技术实现应把数据读写封装成独立层，便于未来替换为云端存储（如 Neon Postgres）而不用重写业务逻辑。

## 10. 验收标准（供测试参考）

| 编号 | 验收场景 | 通过标准 |
|---|---|---|
| AC-1 | 打开应用 | 直接看到首页输入框，无需任何点击即可开始输入 |
| AC-2 | 提交想法 | 提交后出现"它已安静地沉入深处。"反馈文案，输入框清空 |
| AC-3 | 后台生成 | 提交后不出现加载中的等待态阻塞界面；用户稍后进入"深处"页查看该想法时，AI 共情回应已经生成完毕 |
| AC-4 | 查看深处页 | 已提交的所有想法均以节点形式出现在画布上，位置为随机分布且刷新页面后位置保持不变 |
| AC-5 | 查看单条详情 | 点击节点可查看想法原文与 AI 一句话共情回应，回应不含建议/鼓励措辞 |
| AC-6 | 编辑想法 | 可修改想法文字并保存成功，修改后 AI 共情回应重新生成 |
| AC-7 | 删除想法 | 可删除想法，删除后节点从深处页消失，且不可恢复 |
| AC-8 | 聚类效果 | 新增想法后，画布上节点的分组着色会相应更新（同语义倾向的想法着色应趋于一致，允许一定误判） |
| AC-9 | 视觉基调 | 全程无高饱和撞色、无鼓励式弹窗、无进度条/勋章等激励型视觉元素 |
| AC-10 | 数据持久化 | 关闭浏览器/刷新页面后，本地已记录的想法与其 AI 回应、分组均保持不丢失（在同一浏览器同一设备上） |

## 11. 待确认 / 开放问题

以下问题不影响 MVP 启动开发，但建议在开发过程中或迭代中进一步明确：

1. **聚类技术路线**：采用 6.4.2 中方案 1（LLM 归纳打标）还是方案 2（独立 embedding 服务）？建议 MVP 先用方案 1。
2. **共情回应的具体 Prompt 与语气打磨**：需要通过实际生成结果多轮调优，确保不"滑向"建议或鼓励语气。
3. **深处页节点视觉细节**：节点大小/亮度是否需要随时间产生变化（如越"老"的想法光晕越柔和），目前未定义，可作为后续迭代点。
4. **多设备与分享能力**：MVP 明确不做，但数据结构与存储层设计需为后续迁移预留空间。
