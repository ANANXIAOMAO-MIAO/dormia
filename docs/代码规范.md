# Dormia — 代码规范

版本：v0.1（MVP）
关联文档：[PRD.md](../PRD.md)、[技术设计文档.md](技术设计文档.md)
最后更新：2026-07-05

> 本规范用于统一 Dormia 的代码风格与工程实践。目标不是繁琐约束，而是让「静默、克制、可迁移」的设计原则在代码里也保持一致。规范以「能被工具自动执行」为优先（ESLint / Prettier / TS strict）。

---

## 1. 总则

1. 所有代码使用 **TypeScript**，开启 `strict`。
2. **UI 层不直接访问存储或网络**：组件通过 `store` 触发动作，`store` 通过 `IdeaRepository` / `deepseekClient` 完成读写与请求。
3. **静默失败**：AI/网络失败不弹错误、不打断用户；降级到「回应为空 / 默认颜色」。
4. **文案不硬编码在组件里**，一律从 `src/content/copy.ts` 引用。
5. **颜色、间距、动效时长不硬编码**，一律用 `styles/tokens.css` 的 CSS 变量。
6. 提交前必须通过 `lint` 与 `type-check`，无 error。

---

## 2. TypeScript 规范

- `tsconfig` 开启：`strict`、`noUnusedLocals`、`noUnusedParameters`、`noImplicitReturns`、`noFallthroughCasesInSwitch`。
- **禁止随手 `any`**。确需未知类型用 `unknown` 并做收窄；第三方缺类型时局部 `// @ts-expect-error` 并注明原因。
- 数据结构统一用 `interface`（对象）/ `type`（联合、工具类型）。核心类型集中在 `src/data/types.ts`。
- 外部数据（`localStorage` 解析、API 响应）视为 `unknown`，解析后做类型校验或防御式回退，不直接断言。
- 优先不可变数据：更新用展开/`map`/`filter` 生成新对象，避免原地 `mutate`。
- 时间统一用 `number`（epoch ms），不混用 `Date` 与字符串。

---

## 3. 命名规范

| 对象 | 约定 | 示例 |
|---|---|---|
| 组件文件与组件 | PascalCase | `GlowCanvas.tsx` → `GlowCanvas` |
| Hook | `use` 前缀 + camelCase | `useIdeasStore` |
| 普通模块文件 | camelCase | `deepseekClient.ts`、`palette.ts` |
| 类型 / 接口 | PascalCase，不加 `I` 前缀 | `Idea`、`StorageAdapter` |
| 变量 / 函数 | camelCase | `addIdea`、`existingClusters` |
| 常量集合 | camelCase 对象 + `as const` | `copy`、`tokens` |
| CSS Module 类名 | camelCase | `.glowNode`、`.detailCard` |
| CSS 变量 | `--` + kebab-case，带前缀 | `--color-bg`、`--motion-slow` |

命名语气也遵循产品基调：避免任务型词汇（如 `submit`/`create` 面向用户文案层面），但代码内部函数名以清晰为准（`addIdea` 可以）。

---

## 4. 目录与分层职责

严格单向依赖：`pages / components → store → data (repository) / ai (client)`。

| 目录 | 职责 | 禁止 |
|---|---|---|
| `pages/` | 页面组装 | 直接调 `localStorage` / `fetch` |
| `components/` | 纯展示 + 交互，尽量无副作用 | 直接持久化 / 调 AI |
| `store/` | 应用状态与动作编排 | 直接操作 DOM |
| `data/` | 存储抽象与持久化 | 依赖 UI / 调 AI |
| `ai/` | DeepSeek 调用与结果解析 | 直接写存储（交回 store 编排） |
| `lib/` | 纯工具函数（id、position、palette） | 有副作用 / 依赖 store |
| `content/` | 文案常量 | 逻辑 |
| `styles/` | 令牌与全局样式 | — |

- `data/` 只 import `types` 与 `StorageAdapter`；业务只 import `IdeaRepository`，永不直接 import `localStorage` 相关实现。
- `ai/` 只经 `deepseekClient` 发请求，`empathy.ts` / `clustering.ts` 负责构造入参与解析出参。

## 5. React 组件规范

- 一律使用**函数组件 + Hooks**，不用 class 组件。
- Props 显式定义类型：`interface XxxProps { ... }`，避免 inline 匿名类型堆叠。
- 组件保持单一职责；文件超过约 200 行考虑拆分。
- 副作用放 `useEffect`，标注依赖数组；避免在渲染期做 I/O。
- 事件处理函数命名 `handleXxx`（如 `handleSubmit`）。
- 列表渲染提供稳定 `key`（用 `idea.id`，不用数组下标）。
- Canvas / 动画组件用 `requestAnimationFrame`，在 `useEffect` cleanup 中取消，避免泄漏。
- 派生数据用 `useMemo`，稳定回调用 `useCallback`——仅在有实际性能/引用需要时使用，不盲目包裹。

---

## 6. 样式规范

- 使用 **CSS Modules**（`*.module.css`）+ `styles/tokens.css` 的 CSS 变量。
- **禁止硬编码颜色/间距/动效时长**，一律引用令牌：

```css
/* 反例 */
.node { box-shadow: 0 0 24px #6b5bd2; transition: all 0.2s ease; }

/* 正例 */
.node {
  box-shadow: 0 0 var(--glow-radius) var(--color-glow);
  transition: opacity var(--motion-slow) var(--ease-soft);
}
```

- 动效遵循产品基调：**缓慢、柔和**；禁止 bounce、快速缩放等轻快效果。
- 暗色蓝紫冷色系；禁止高饱和撞色、进度条、勋章、彩带等激励型视觉元素。
- 布局保持大量留白，字体简洁，不滥用强调色。

---

## 7. AI / 网络调用规范

- 所有 DeepSeek 调用经 `src/ai/deepseekClient.ts`，前端**绝不**直接持有或调用带 Key 的 DeepSeek 接口，一律走 `/api/deepseek` 代理。
- 请求设**超时**（建议 10s）；可对网络抖动做**一次**轻量重试，仍失败即降级。
- **静默失败**：捕获异常后不 `throw` 到 UI、不弹提示；empathy 失败 → 保留空回应，cluster 失败 → `clusterId` 保持 `null`。
- AI 为 **fire-and-forget**：`addIdea` 必须先完成持久化并让 UI 返回，AI 结果后续再回填，绝不阻塞用户输入（对齐 AC-3）。
- 响应体按 `unknown` 处理并做字段校验后再使用，防止脏数据污染 store。
- Prompt 主体保留在服务端（`api/deepseek.ts`）；前端只传 `ideaText` / `existingClusters` 等数据。

---

## 8. 文案规范

- 所有用户可见文字来自 `src/content/copy.ts`，禁止散落在组件中的中文字面量（占位符、aria-label 也尽量集中）。
- 语气统一：**冷静、克制、不评判、不鼓励行动**。
- 禁止出现：`加油`、`你可以的`、`很棒`、`快去…吧`、`确定删除吗？此操作不可撤销` 等鼓励式 / 警告式措辞。
- 删除等操作用轻表达（如「它将不再被记起。」），不用严肃警告。
- AI 生成文案的语气约束写在服务端 Prompt 中，评审时对照 PRD 6.4 正反例。

## 9. 工程规范

### 9.1 Lint / 格式化
- **ESLint**：`@typescript-eslint` + `eslint-plugin-react-hooks`；开启 `react-hooks/rules-of-hooks`、`react-hooks/exhaustive-deps`。
- **Prettier** 统一格式：单引号、尾逗号 `all`、`semi: true`、`printWidth: 100`。
- 提交前 `pnpm lint && pnpm type-check`（或对应 npm/yarn 脚本）需通过，无 error。可选接入 `lint-staged` + Git hook。

### 9.2 Import 顺序
1. 第三方库（react、zustand…）
2. 内部绝对/别名模块（`@/data`、`@/ai`…）
3. 相对模块（`./`、`../`）
4. 样式（`*.module.css`）放最后

每组之间空一行；同组按字母序。

### 9.3 注释准则
- 只写「为什么」，不写「做了什么」的复述性注释。
- 记录非显而易见的约束/取舍（如「坐标归一化以避免刷新位置跳动」）。
- 禁止提交被注释掉的死代码。

### 9.4 环境变量与密钥
- `DEEPSEEK_API_KEY` 只存部署平台环境变量与本地 `.env`（`.gitignore` 忽略），**绝不进代码库**。
- 仓库仅保留 `.env.example`（无真值）。
- 前端环境变量（`VITE_` 前缀）不得放任何密钥。

### 9.5 Git commit 约定
- 采用 Conventional Commits：`type(scope): subject`。
- 常用 `type`：`feat`、`fix`、`refactor`、`style`、`docs`、`chore`、`build`。
- 示例：
  - `feat(home): 首页记录闭环与本地持久化`
  - `feat(ai): 接入 DeepSeek 共情回应代理`
  - `refactor(data): 抽象 StorageAdapter 以便云端迁移`
- subject 用中文简述，聚焦意图；一次提交只做一件事。

## 10. 检查清单（提交前自检）

- [ ] 无 TS error，`strict` 通过，无随手 `any`。
- [ ] UI 未直接碰 `localStorage` / `fetch`，走 store/repo/ai。
- [ ] 用户可见文字来自 `copy.ts`，语气克制无鼓励/警告措辞。
- [ ] 颜色/间距/动效用令牌，无硬编码；无激励型视觉元素。
- [ ] AI 调用静默失败、fire-and-forget、不阻塞输入。
- [ ] 无密钥进库；`.env` 已忽略。
- [ ] Lint / Prettier 通过；commit 符合约定。
